## 一、今日最值得关注的 3 条内容

### 1. Anthropic Economic Index report: Cadences
- 来源链接：[Anthropic 官方报告](https://www.anthropic.com/research/economic-index-june-2026-report)；[官方 PDF](https://cdn.sanity.io/files/4zrzovbb/website/9e0eadc8097864886c5d5060ebb1f89b02ea29d6.pdf)；本地文件：[outputs/sources/anthropic-economic-index-cadences-2026-06-26.pdf](sources/anthropic-economic-index-cadences-2026-06-26.pdf)
- 类型：研究报告
- 推荐指数：5/5 星
- 一句话总结：Anthropic 用更细粒度的 Claude 使用数据说明，AI 使用正在从聊天问答转向可委派的 artifact production 和 long-running agentic tasks。
- 为什么重要：这份报告对 AI 产品经理很有价值，因为它不只讨论“AI 能不能做”，而是在拆用户何时用、产出什么、交给 AI 多大自主权、使用者如何感知未来 12 个月的工作变化。它把 Agent 产品判断从主观体验拉回到数据指标。
- 核心观点 3 条：
  1. Claude 使用形态正在变化：报告明确说，随着 Claude Code 和 Cowork 增长，很多 session 已经不再是简单 chat transcript，而是 long-running agentic tasks。
  2. 产品形态比模型本身更关键：同样是生成代码、脚本、网站等产物，Claude Code 上的 AI autonomy 明显高于 chat/Cowork；即使用同一模型对比，差异仍然存在。
  3. 高价值工作消耗更多 compute：报告把 artifact、token consumption、wage proxy 放在一起看，发现 apps/websites 等复杂产物消耗远高于中位数；自动化使用更高的人更乐观看待 AI 对技能价值的影响，但报告也提醒不能排除 skill erosion。
- 对 AI 产品经理 / FDE 的启发：以后判断一个 AI 产品，不要只问“模型多强”，要问它能稳定产生什么 artifact、用户愿意委派多少判断、是否能保留人工复核和上下文。
- 可落地行动或作品集切入点：做一个“AI 产品工作流度量表”，按 artifact 类型、AI autonomy、人工参与轮次、验证方式、token/cost、失败风险来评估你自己的 Agent 项目。
- 风险提醒：报告基于隐私保护后的 Anthropic 自有数据，不等于全市场；自动化使用者更乐观可能有自选择偏差。
- 是否值得精读：是。重点读 Introduction、Artifacts、Perceptions 三部分。

### 2. Previewing GPT-5.6 Sol: a next-generation model
- 来源链接：[OpenAI 官方页面](https://openai.com/index/previewing-gpt-5-6-sol/)；[GPT-5.6 Preview System Card](https://deploymentsafety.openai.com/gpt-5-6-preview)；[官方 PDF](https://deploymentsafety.openai.com/gpt-5-6-preview/gpt-5-6-preview.pdf)；本地文件：[outputs/sources/openai-gpt-5-6-preview-system-card-2026-06-26.pdf](sources/openai-gpt-5-6-preview-system-card-2026-06-26.pdf)
- 类型：官方博客 / 系统卡
- 推荐指数：5/5 星
- 一句话总结：OpenAI 预览 GPT-5.6 Sol/Terra/Luna，重点不是“又一个模型”，而是 reasoning effort、subagents、agentic coding eval 和分层安全发布机制正在成为新产品基建。
- 为什么重要：这条对未来 3-12 个月 AI 产品形态有信号意义。OpenAI 把 `max` reasoning effort、`ultra` subagent mode、Terminal-Bench 2.1、cyber/bio safeguards、trust-based access 放在同一套发布叙事里，说明高阶 Agent 能力会越来越依赖“能力档位 + 安全栈 + 访问分层 + eval”。
- 核心观点 3 条：
  1. GPT-5.6 不是单一模型，而是 Sol、Terra、Luna 三档；Sol 先小范围 trusted preview，再计划后续更广泛开放。
  2. 新能力方向明显 agentic：`max` 给复杂任务更多推理时间，`ultra` 通过 subagents 加速复杂工作，编码评估强调终端工作流、规划、迭代和 tool coordination。
  3. 系统卡把安全做成产品层：包括模型内拒答、实时分类器、账号级风险信号、trust-based access、自动红队、computer use 的用户确认，以及 data-destructive action 防护。
- 对 AI 产品经理 / FDE 的启发：未来的企业 Agent 产品不能只做“开始执行”按钮，要设计任务难度档位、权限边界、成本预算、人工确认、日志追踪和异常中止。
- 可落地行动或作品集切入点：做一个 Agent Run Control 面板原型：用户可选 reasoning effort、预算上限、允许工具、敏感操作需确认、输出带 trace 和 eval 结果。
- 风险提醒：OpenAI 主页面在 PowerShell 校验时返回 403，但网页检索可读，官方 PDF 可正常下载；模型仍是 preview，不应把能力当作已全面可用。
- 是否值得精读：是。重点读官方页面的 Capabilities 和系统卡 Introduction / Safeguards。

### 3. How Notion used the Cursor SDK to embed coding agents
- 来源链接：[Cursor 官方博客](https://cursor.com/blog/notion)
- 类型：博客 / 工程案例
- 推荐指数：4/5 星
- 一句话总结：Notion 把 Cursor SDK 嵌进自己的协作产品，让用户在 doc、thread、database issue 里直接委派 coding agent，最终产出 PR。
- 为什么重要：这是 Agent 产品形态的好样本：Agent 不再只是 IDE 里和工程师聊天，而是进入团队协作表面。Notion 负责 context 和 surface，Cursor 负责 agent engine、sandbox、model routing、tool use、remote MCP。
- 核心观点 3 条：
  1. 产品集成思路是 thin adapter：Notion thread 映射为 Cursor agent，每条消息映射为一次 agent run。
  2. 用户体验要能观察和恢复：后续 run 通过 SSE stream 给用户看进度，连接断开后可以从最后事件恢复。
  3. 真正可用依赖上下文和工具：通过 remote MCP，Cursor 可读写 Notion workspace，并按模板、skills、subagents、自定义触发器来适配任务。
- 对 AI 产品经理 / FDE 的启发：好 Agent 不是一个孤立窗口，而是“原工作流中的委派入口 + 可见执行过程 + PR/变更物 + 人工验收”。
- 可落地行动或作品集切入点：做一个 mini FDE demo：从 Notion/Feishu/TAPD issue 触发 Agent，自动生成 plan、代码变更、测试结果和人工验收清单。
- 风险提醒：这类嵌入式 Agent 必须处理 repo 权限、MCP 权限、PR attribution、失败恢复和敏感数据边界。
- 是否值得精读：是。需上线阅读全文。

## 三、值得深入学习的开源项目

### omnigent-ai/omnigent
- GitHub 链接：[omnigent-ai/omnigent](https://github.com/omnigent-ai/omnigent)
- Star 数：5,134
- 解决什么问题：做 Claude Code、Codex、Cursor、Pi、自定义 agents 之上的 meta-harness，统一编排、策略、安全沙箱、成本控制和多人协作。
- 为什么值得我学习：它抓住了今天的主线：未来不是单 Agent，而是多 harness、多模型、多权限、多会话的管理层。
- 如何用到我的项目 / 工作中：可以参考它做一个“Agent Command Center”作品集，把任务拆分、预算上限、人工审批、会话共享、产出复核做成可演示流程。延伸阅读：[Databricks Omnigent 博客](https://www.databricks.com/blog/introducing-omnigent-meta-harness-combine-control-and-share-your-agents)、[Latent Space Meta-Harness Summer](https://www.latent.space/p/ainews-its-meta-harness-summer)。

### vercel/eve
- GitHub 链接：[vercel/eve](https://github.com/vercel/eve)
- Star 数：2,804
- 解决什么问题：filesystem-first durable AI agents，把 instructions、tools、skills、channels、schedules 放进固定目录，让 agent 更容易被检查、扩展和运行。
- 为什么值得我学习：它非常适合理解“Agent 不是 prompt，而是 repo-native workflow primitive”。
- 如何用到我的项目 / 工作中：可以用它做一个每日信息捕获 agent 或求职作品集 agent，把定时任务、工具调用、人工确认、输出文件都放进项目目录。

### agentclientprotocol/agent-client-protocol
- GitHub 链接：[agentclientprotocol/agent-client-protocol](https://github.com/agentclientprotocol/agent-client-protocol)
- Star 数：3,515
- 解决什么问题：标准化 editor/client 与 coding agent 的通信，让不同 IDE 和不同 coding agent 可以互操作。
- 为什么值得我学习：ACP 和 MCP 一样，是理解 Agent 工程生态的重要协议层；一个连工具，一个连客户端/编辑器。
- 如何用到我的项目 / 工作中：作品集可以做一页“Agent interoperability map”，解释 MCP、ACP、harness、sandbox、PR review 分别负责什么。
