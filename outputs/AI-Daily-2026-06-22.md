## 一、今日最值得关注的 3 条内容

### 1. OpenAI：Predicting model behavior before release by simulating deployment
- 来源链接：[OpenAI 官方博客](https://openai.com/index/deployment-simulation/) / [官方 PDF](https://cdn.openai.com/pdf/predicting-llm-safety-before-release-by-simulating-deployment.pdf) / 本地文件：[outputs/sources/openai-deployment-simulation-2026-06-16.pdf](sources/openai-deployment-simulation-2026-06-16.pdf)
- 类型：论文 / 官方博客
- 推荐指数：★★★★★
- 一句话总结：OpenAI 提出 Deployment Simulation，用脱敏真实对话前缀重放候选模型，提前预测上线后的不良行为、eval awareness 和工具调用风险。
- 为什么重要：Agent 产品最难的不是 demo，而是上线后在真实流量、真实工具、真实权限下是否稳定。这个方法把 eval 从“写题目考模型”推进到“模拟真实部署分布”，对 FDE 做客户场景验证非常有参考价值。
- 核心观点 3 条：
  1. 传统 eval 容易覆盖窄、样本偏、被模型识别为测试；Deployment Simulation 用真实分布前缀降低这些问题。
  2. OpenAI 在约 130 万条脱敏对话上做实验，并用 taxonomy coverage、directional accuracy、rate calibration 验证预测质量。
  3. Agentic tool use 的模拟难点在外部状态和读写工具真实性，论文明确说它是补充，不替代红队、对抗评测和尾部风险分析。
- 对 AI 产品经理 / FDE 的启发：做企业 Agent 时，不要只写 20 个理想 prompt，要抽样真实但脱敏的工单、企微 SOP、客户运营任务，重放候选模型并记录失败率、越权倾向、幻觉、工具误用。
- 可落地行动或作品集切入点：做一个“客户运营 Agent 发布前评测 harness”：输入脱敏任务前缀，跑新旧模型，对比触达文案、标签更新、群发建议、链接校验、人工确认点，输出 release gate。
- 风险提醒：生产数据必须先脱敏并确认授权；不要上传客户姓名、手机号、企微聊天、合同、订单、密钥；工具模拟不能直接连生产系统，先用 sandbox 或 mock。
- 是否值得精读：是。

### 2. Google：ARD + A2A 正在补齐 agentic web 的发现、信任和协作层
- 来源链接：[Agentic Resource Discovery](https://developers.googleblog.com/announcing-the-agentic-resource-discovery-specification/) / [A2A collaborative agents](https://developers.googleblog.com/how-a2a-is-building-a-world-of-collaborative-agents/) / [A2UI](https://developers.googleblog.com/introducing-a2ui-an-open-project-for-agent-driven-interfaces/)
- 类型：官方博客 / 开放规范
- 推荐指数：★★★★★
- 一句话总结：Google 把 Agent 生态拆成三层：ARD 负责发现能力，A2A 负责 agent-to-agent 协作，A2UI 负责跨端生成式 UI 表达。
- 为什么重要：Agent 产品正在从单点工具调用变成跨组织、跨协议、跨权限边界的协作网络。FDE 未来交付的不是“一个 bot”，而是一套可发现、可验证、可治理的能力目录。
- 核心观点 3 条：
  1. ARD 用 domain-hosted catalog 和 federated registry，让 agent 找到 MCP server、A2A agent、OpenAPI tool 等能力，并带上信任元数据。
  2. A2A 强调 secure boundary 和 zero context pollution：敏感流程留在专用 agent 内部，主 agent 只拿结果，减少上下文污染和数据暴露。
  3. A2UI 把 agent 返回的 UI 从纯文本推进到可移植组件蓝图，适合多 agent、多前端、多企业系统协作。
- 对 AI 产品经理 / FDE 的启发：企业 Agent 方案要开始设计“能力目录 + 权限边界 + 审计记录”，而不是把所有工具塞进一个万能 prompt。尤其在企微客户运营里，标签、群发、CRM、知识库、工单应拆成可控能力。
- 可落地行动或作品集切入点：做一个“企微 Agent Capability Catalog”原型：列出客户标签查询、SOP 推荐、群发草稿、链接检测、数据看板等能力，每个能力写用途、输入输出、权限、风险等级、人工确认规则。
- 风险提醒：能力发现不等于自动授权；跨 agent 调用要防 prompt injection、越权读取、错误群发、错误写库；catalog 里不能暴露内部接口密钥和客户数据样例。
- 是否值得精读：是。

### 3. Dwarkesh：The data black hole at the center of AI
- 来源链接：[Dwarkesh Podcast / Essay](https://www.dwarkesh.com/p/the-sample-efficiency-black-hole)
- 类型：长文 / 访谈延伸
- 推荐指数：★★★★☆
- 一句话总结：Dwarkesh 用 sample efficiency 解释当前模型的边界：AI 可以通过海量训练把常见白领任务纳入分布，但未必已具备人类式快速学习。
- 为什么重要：这能帮助 AI PM 判断哪些流程适合自动化，哪些流程还需要人类专家兜底。对作品集也很关键：不要声称 Agent 能“理解一切”，要证明它能在明确分布、明确反馈、明确 eval 下稳定工作。
- 核心观点 3 条：
  1. 近期能力提升更像是扩大和改善训练数据分布，而不是已经解决人类级 sample efficiency。
  2. 常见白领任务可以通过 RL、SFT、verifier 和合成数据进入模型分布，因此仍有巨大商业价值。
  3. 真正稀缺的是 out-of-distribution 判断、意图选择、问题定义和持续学习，这些仍需要专家与组织流程。
- 对 AI 产品经理 / FDE 的启发：选场景时优先找“高频、可验证、可复用、低隐私暴露”的任务，比如客户分层、活动复盘、知识库问答、跟进提醒；谨慎承诺低频复杂决策的全自动化。
- 可落地行动或作品集切入点：把作品集里的企微客户运营 Agent 分成 in-distribution 与 OOD 两栏：前者自动跑，后者触发人工审核，并沉淀失败样例进入 eval set。
- 风险提醒：长文/访谈需上线阅读全文/收听；这是个人判断，不是官方研究结论。引用时不要把“sample efficiency 仍是瓶颈”扩展成“AI 自动化没有价值”。
- 是否值得精读：是。

## 二、今日最重要的一个观点

> 今天我最应该记住的是：Agent 的竞争力正在从“模型回答好不好”转向“真实工作流能否被发现、授权、模拟、评测、审计和人工验收”。AI PM / FDE 的核心价值，是把业务场景变成可复用的能力目录、eval harness 和安全发布流程。

### ChromeDevTools/chrome-devtools-mcp
- GitHub 链接：[ChromeDevTools/chrome-devtools-mcp](https://github.com/ChromeDevTools/chrome-devtools-mcp)
- Star 数：44,138（GitHub API，2026-06-22 核验）
- 解决什么问题：让 coding agent 通过 MCP 控制和检查真实 Chrome，读取 console/network/performance trace，做前端调试和性能分析。
- 为什么值得我学习：它代表“工具使用从读文件走向真实应用环境”的方向，适合理解 Agent 如何获得可验证的外部反馈。
