## 一、今日最值得关注的 3 条内容

### 1. Anthropic: Introducing Claude Tag
- 来源链接：[Anthropic 官方博客](https://www.anthropic.com/news/introducing-claude-tag)
- 类型：官方博客 / Agent 产品形态
- 推荐指数：★★★★★
- 一句话总结：Claude Tag 把 Agent 从个人聊天窗口推进到 Slack 频道里的“团队协作成员”，重点能力是 channel-scoped context、memory、tools、async work 和权限审计。
- 为什么重要：这不是简单的 Slack bot，而是企业 Agent 产品形态的一次清晰示范。Claude 可以被授权进入指定频道、连接指定工具和代码库、记住频道相关信息，并在被 @ 后异步执行任务。Anthropic 还提到内部产品团队有大量代码由内部版 Claude Tag 创建，这说明团队级 Agent 正在从“个人增效工具”变成“组织工作流节点”。
- 核心观点 3 条：
  1. Agent 需要“多人共享上下文”。同一个频道里的 Claude 可以让团队成员接续对话，不再每个人都从零解释背景。
  2. Memory 必须按频道和用途隔离。销售、工程、支持等不同身份的 Claude 应该有不同工具、数据和记忆边界。
  3. Async 与 ambient behavior 是下一步。Agent 不只被动回答，也可以跟进沉默线程、提示相关信息、安排未来任务，但必须被权限和日志约束。
- 对 AI 产品经理 / FDE 的启发：企微场景可以借鉴这个形态，但要更保守。与其做一个“万能企微机器人”，不如做多个 channel-scoped / role-scoped Agent，例如客户运营助手、售后跟进助手、素材质检助手。每个 Agent 的可见客户数据、可调用工具、可写回字段和可触发群发动作都要分开定义。
- 可落地行动或作品集切入点：做一个“企微 @AI 客户运营助手”原型设计：频道级权限、客户资料读取范围、素材生成、链接校验、敏感词/隐私检查、群发草稿、人工确认、操作日志。作品集重点展示权限矩阵和审计流，而不是只展示聊天 UI。
- 风险提醒：需上线阅读全文。不要把真实客户聊天、手机号、订单、合同、企微联系人或内部截图接入 demo；主动提醒和异步跟进容易制造误触达，群发、写库、外链发送必须默认人工确认。管理员还要关注 token spend、日志可见性和不同频道之间的记忆隔离。
- 是否值得精读：是。尤其值得看 “Working with @Claude” 和 “Getting started” 两节。

### 2. OpenAI: Codex-maxxing for long-running work
- 来源链接：[OpenAI 官方页面](https://openai.com/index/codex-maxxing-long-running-work/) / [官方 PDF](https://cdn.openai.com/pdf/8a9f00cf-d379-4e20-b06f-dd7ba5196a11/OAI_WhitePaper_Codex-maxxing26.pdf) / 本地文件：[outputs/sources/openai-codex-maxxing-long-running-work-2026-06-22.pdf](sources/openai-codex-maxxing-long-running-work-2026-06-22.pdf)
- 类型：官方白皮书 / Workflow / Loop Engineering
- 推荐指数：★★★★★
- 一句话总结：这份白皮书把 Codex 描述成长期工作空间，不只是一次性 coding assistant，核心是 durable threads、steering、memory、tools、thread automations 和 verifiable goals。
- 为什么重要：AI PM / FDE 的竞争力会越来越像“工作流设计师”。单次 prompt 能产出内容，但真正的高杠杆来自让任务能持续、可复查、可接力、可验证。OpenAI 这份材料把 long-running work 拆成一组可以复用的工作部件，非常适合转成个人作品集方法论。
- 核心观点 3 条：
  1. Durable thread 是工作容器。重要项目要有可回来的地方，让历史决策、偏好、开放问题和下一步动作持续累积。
  2. Memory 要可打开、可编辑、可 diff。不要让长期记忆静默藏在聊天记录里，应该沉淀成人、项目、决策、闭环状态等可审查资产。
  3. 强目标要可验证。好任务不是“按计划实现”，而是给出测试、验收标准、约束和 definition of done，让 Agent 知道什么时候才算完成。
- 对 AI 产品经理 / FDE 的启发：这正好对应用户正在做的 AI Daily 自动化。每天不是重新开一个 prompt，而是让 Agent 带着记忆、来源口径、下载规则、公开风险检查和 git 上传流程持续运行。以后做企微客户运营 Agent，也应该先定义 loop：捕获信号、生成草稿、校验链接、拦截隐私、人工确认、记录结果。
- 可落地行动或作品集切入点：把“AI Daily 信息捕获”本身包装成作品集案例：输入源、筛选规则、memory.md、PDF source archive、人工验证清单、git public publishing、失败处理。再迁移到“企微客户运营 Follow-up Loop”，用 mock 客户数据演示从客户意图到草稿再到人工确认的闭环。
- 风险提醒：PDF 已下载，本地大小约 3.56 MB，头部为 `%PDF`，仍建议人工打开复核。长期线程和记忆很容易积累敏感数据，尤其是 Slack、Gmail、Chrome、企微、CRM 这类连接器；任何外发邮件、群发消息、客户资料导出、写库动作都必须保留人工确认点。
- 是否值得精读：是。优先读 Durable threads、Memory、Thread automation、Goals 四部分。

### 3. Human-on-the-Bridge: Scalable Evaluation for AI Agents
- 来源链接：[arXiv 摘要](https://arxiv.org/abs/2606.16871) / [arXiv PDF](https://arxiv.org/pdf/2606.16871) / 本地文件：[outputs/sources/human-on-the-bridge-agent-evaluation-2606.16871v1.pdf](sources/human-on-the-bridge-agent-evaluation-2606.16871v1.pdf)
- 类型：论文 / Evaluation / Agent Harness
- 推荐指数：★★★★★
- 一句话总结：Human-on-the-Bridge 认为 Agent 评测要评估“行为系统”，把人类专家判断前置为可复用评测资产，再由 harness 执行多轮对抗、trace capture、多 juror scoring 和 evidence-linked report。
- 为什么重要：很多团队还在用“最终回答看起来对不对”评价 Agent，但 Agent 的真实失败常发生在轨迹里：声称调用了工具但没调用、跳过必需工具、策略漂移、被用户诱导改写规则、拒绝得很安全但没有解决问题。HOB 的价值在于把人工判断从每次手动 review，升级为可重复运行的 evaluation intelligence。
- 核心观点 3 条：
  1. Agent evaluation 不是单轮答案评分，而是 behavioral、procedural、adversarial、evidence-linked assessment。
  2. HOB 的可复用资产包括 domain context、Red-Team Traps、Juror Personas、scoring guidelines、audit rules、fallback policies。
  3. 论文报告覆盖 23,500 个 agent turns，暴露了 phantom tool calls、missing mandatory tool calls、policy drift、manipulation paths 等静态 benchmark 容易漏掉的问题。
- 对 AI 产品经理 / FDE 的启发：作品集里一定要有 eval set 和 trace 证据。对于企微客户运营 Agent，不能只问“回复是否好听”，还要检查是否读取了正确客户上下文、是否跳过隐私检查、是否错误承诺优惠、是否发送未经确认的链接、是否把客户数据写进不该写的地方。
- 可落地行动或作品集切入点：做一个 HOB 风格的“客户运营 Agent Eval Pack”：10 条 Red-Team Traps、3 个 Juror Personas、6 个指标、必需工具调用规则、隐私 fallback policy、trace 证据模板。可尝试参考 [ProofAgent-ai/proofagent-harness](https://github.com/ProofAgent-ai/proofagent-harness)，但先用 mock 数据跑，不接生产企微。
- 风险提醒：这是 arXiv 预印本，需进一步核验，不能当作行业定论；关联开源 harness 当前星标很少，工程成熟度要单独评估。评测样例不要包含真实客户数据、病历、财务、合同、手机号或内部密钥；对抗样例也不要直接拿到生产群里测试。
- 是否值得精读：是。建议重点读 Introduction、The Human-on-the-Bridge Paradigm、Tool Use and Procedural Agent Failures。

## 二、今日最重要的一个观点

> 今天我最应该记住的是：Agent 产品正在从“单次聊天”变成“常驻团队成员 + 长周期工作 loop + 可复用评测资产”。AI 产品经理 / FDE 的核心能力不是会写更花的 prompt，而是能设计上下文、权限、工具、记忆、人工确认、trace 和 eval，让 Agent 在真实业务里可控、可查、可复盘。

## 三、值得深入学习的开源项目

### ProofAgent-ai/proofagent-harness
- GitHub 链接：[ProofAgent-ai/proofagent-harness](https://github.com/ProofAgent-ai/proofagent-harness)
- Star 数：5（GitHub API，2026-06-24 核验）
- 解决什么问题：为 AI agents 提供类似 pytest 的多轮对抗评测 harness，支持 juror panel、production-critical metrics、trace、report 和 CI。
- 为什么值得我学习：星标很低，工程成熟度需进一步核验，但它和 HOB 论文的方法论强相关，适合学习“Agent 评测资产怎么结构化”。
- 如何用到我的项目 / 工作中：先不用真实企微数据，拿 mock 客户运营 Agent 跑退款、群发、隐私泄露、错误链接、客户标签误写等场景。

### headroomlabs-ai/headroom
- GitHub 链接：[headroomlabs-ai/headroom](https://github.com/headroomlabs-ai/headroom)
- Star 数：48,579（GitHub API，2026-06-24 核验）
- 解决什么问题：压缩 tool outputs、logs、files、RAG chunks 和对话历史，提供 library、proxy、MCP server、cross-agent memory 等能力，目标是减少上下文成本。
- 为什么值得我学习：Context Engineering 的痛点越来越明确，Agent 不是上下文越塞越多越好，而是要筛选、压缩、可追回原文。
- 如何用到我的项目 / 工作中：可作为“客户运营知识库 + 工具日志压缩”的学习参考，但不要直接接入含客户隐私的数据；先用脱敏文档验证压缩后是否影响回答质量。

### upstash/context7
- GitHub 链接：[upstash/context7](https://github.com/upstash/context7)
- Star 数：57,936（GitHub API，2026-06-24 核验）
- 解决什么问题：通过 CLI / MCP 把最新、版本相关的库文档放进 LLM context，减少 AI coding 里的过时 API 和 hallucinated APIs。
- 为什么值得我学习：FDE 做 demo 和交付时，经常卡在 SDK 版本、Auth API、框架升级上；Context7 代表一种很实用的“外部文档注入”模式。
- 如何用到我的项目 / 工作中：做作品集技术实现时，可以用它查询 Supabase、Next.js、LangGraph 等最新文档；每次引用外部 API 时保留链接和版本说明。
